# AI Council City — эталонная архитектура (строгий референс)

Этот документ описывает подтверждённую логику работы системы. Любой
AI-исполнитель (Cursor, Codex, Antigravity, Claude), работающий над кодом
этого проекта, должен сверяться с этим документом перед началом задачи.

Если задание касается конкретной функции, отдела кода, файла — работа
строго в этих границах. Запрещено трогать другие файлы, функции, стили,
типы, схему БД или логику, даже если кажется что они связаны с задачей.
Если для выполнения задачи реально необходимо выйти за указанные границы —
исполнитель останавливается и сообщает об этом как блокер с вопросом, а не
делает это самостоятельно.

---

## 1. Структура

**Иерархия:** City → Building → Chamber → Agent.

**City Hall** — главный контейнер города.
- В главном отделе City Hall сидит Mayor (главный агент).
- Внутри City Hall есть отдельный chamber "Отдел советников" — активируется
  только при включённом режиме Turbo.

**Building (здание)** — это проект (любая сфера деятельности: Citizly,
авто-бизнес, контент и так далее).
- У каждого здания есть один главный отдел (chamber с routing_role='main').
- В главном отделе сидит главный агент здания (менеджер).
- Здание само ничего не решает — оно контейнер. Решения принимает менеджер
  главного отдела.

**Chamber (отдел)** — рабочая единица внутри здания.
- Может быть главным (routing_role='main') или обычным внутренним отделом.
- Внутренние отделы определяют менеджер главного отдела того же здания.

**Agent** — конкретный исполнитель внутри отдела.

### Жёсткие границы

- Здание не выходит за свои рамки без connection.
- Отдел не выходит за свои рамки без connection.
- Агенты не взаимодействуют друг с другом без connection.

---

## 2. Connections (кабели)

Connection — это не визуальная декорация, а реальная запись в БД: контракт
доступа между двумя сущностями.

- Связывают главный отдел одного здания с главным отделом другого здания.
- Также могут связывать отделы напрямую (не только главные).
- Несут права доступа (read_knowledge, read_rules, read_results,
  send_tasks).

---

## 3. Тарифы (режимы активации агентов)

Режим — это фильтр по cost_tier ВНУТРИ уже выбранного отдела, не лимит на
число агентов. В ростере отдела четыре уровня: **free → cheap → mid → premium**
(см. `lib/cost-tier.ts`, `lib/agent-selection.ts`).

- **Fast** — все агенты отдела с cost_tier = **free**.
- **Team** — все **free + cheap** (требуется хотя бы один cheap-агент в ростере).
- **Council** — все **free + cheap + mid** (требуется хотя бы один mid-агент).
- **Turbo** — все **четыре уровня** (free + cheap + mid + premium); требуется
  premium-агент. Отдельный независимый режим (не «уровень тарифа» поверх Council),
  активирует топовые модели. Технически реализован как отдельный фиксированный
  chamber «Отдел советников» внутри City Hall: при включении запрос параллельно
  уходит в целевой отдел и в отдел советников, Mayor синтезирует оба ответа.

Сколько агентов реально включается в работу — зависит от того, сколько их
физически привязано к этому отделу на каждом уровне (через
agent_assignments). Если агентов нужного уровня нет — должна быть явная
ошибка, никакого тихого fallback на другой уровень.

## 4. Путь обычного запроса

```
Пользователь → Mayor
  → Mayor определяет нужное ЗДАНИЕ (по названию/описанию задачи)
    → запрос уходит в главный отдел этого здания
      → главный агент (менеджер) здания определяет, какой ВНУТРЕННИЙ
        отдел должен заниматься этим вопросом (тем же принципом, каким
        Mayor выбирал здание)
        → все агенты этого внутреннего отдела (по активному тарифу)
          работают над вопросом параллельно, каждый выдаёт свою версию
        → главный агент здания собирает все версии, синтезирует в один
          развёрнутый ответ
      → развёрнутый ответ возвращается Mayor
  → Mayor сжимает ответ до адекватного размера (если не запрошен полный)
  → ответ выдаётся пользователю
```

Важно: Mayor работает только на уровне здания. Он не выбирает chamber
внутри чужого здания самостоятельно — это исключительно ответственность
менеджера главного отдела того здания.

### Подтверждено по факту (проверено на живых данных)

- Explicit_name роутинг (когда здание названо прямо в запросе) — работает,
  routing_logs записывает delegated_building_id/delegated_chamber_id из
  самого decision-объекта.

### Не подтверждено, требует проверки

- Semantic-роутинг (когда здание не названо явно, Mayor должен понять цель
  по смыслу) — не проверен на практике.
- Реальный механизм, определяющий итоговый agent_count — найдены случаи,
  где запросы попадали в chamber без единой строки в agent_assignments, и
  непонятно откуда в этом случае взялось число агентов. Требуется проверка
  кода, не предположение.

---

## 5. Запрос аналитики/данных (не только задачи)

Mayor может запросить у главного отдела здания любые накопленные данные —
не только выполнить задачу, но и получить статус: какие агенты работали над
конкретным вопросом, сколько агентов сейчас в отделе, сколько агентов во
всём здании, и так далее.

Запрос идёт тем же путём, что обычная задача — через главный отдел здания.
Mayor не обращается к внутренним отделам напрямую. У Mayor должна быть
возможность получить любые данные, аккумулированные в здании и его
отделах, но строго посредником через главный отдел.

## 6. Архивы

Единый механизм сжатия применяется на каждом уровне иерархии, с разной
глубиной детализации.

### 6.1 Архив отдела (chamber_archive)

- **Свежий слой (raw)** — последние записи (договорённость: 5 последних)
  хранятся как есть, не архивируются.
- **Сжатый слой (summary)** — при превышении порога объёма ИЛИ при
  недельном простое без новых raw-записей старые raw сжимаются в один
  абзац через LLM-вызов.
- Сжатие кумулятивное: `new_summary = compress(old_summary + все новые raw
  с последнего обновления)`. Старая summary всегда участвует в новом сжатии
  — иначе раннюя история теряется навсегда после первого цикла сжатия.
- Старые raw-записи физически не удаляются, помечаются полем
  `archived_into`, указывающим на новую summary-запись.

### 6.2 Архив здания (building_archive)

- Пишется ТОЛЬКО менеджером главного отдела здания, по расписанию (раз в
  1-2 дня), не в момент запроса.
- Собирает summary каждого внутреннего chamber_archive здания в один
  building-level дайджест.
- Mayor читает ТОЛЬКО уже готовый building_archive — никогда не лезет в
  chamber_archive напрямую, и не запускает сжатие сам.

### 6.3 Память Mayor (диалог + user_portrait)

- Для пользователя не существует разделения "это решалось через Mayor / это
  решалось напрямую в отделе" — в готовом продукте Mayor отвечает за всё,
  никогда не говорит "я не в курсе, я этого не делал".
- У Mayor есть короткая память диалога: свежий слой (последние сообщения за
  период, дословно) + сжатый слой (старое схлопывается до уровня "был такой
  запрос, ушёл в такое здание, дата").
- Дополнительно Mayor ведёт **user_portrait** — отдельную структурированную
  картину пользователя, обновляется на регулярной основе (например раз в
  несколько дней) из обоих источников: mayor_conversation_log +
  building_archive. Это позволяет Mayor отвечать на вопросы о прошлой работе
  даже если тот разговор шёл напрямую с отделом, минуя Mayor.
- Точный механизм прунинга user_portrait (что делать, когда фактов
  накопится слишком много) не финализирован — открытый вопрос.

### 6.4 Поиск по архиву

- Если нужный ответ есть в свежем слое того уровня, к которому обратились —
  отдаётся сразу, без похода вглубь иерархии.
- Если информация уже выпала в сжатый слой и точные детали утеряны на этом
  уровне — идёт повторный запрос вниз по цепочке (Mayor → главный отдел
  здания → конкретный внутренний отдел) до уровня, где сохранилась нужная
  глубина.
- Поиск в сжатом слое — по ключевым словам.

---

## 7. Открытые вопросы (не реализовано / не подтверждено)

- Semantic-роутинг без явного названия здания — не проверен на практике.
- Реальный механизм формирования agent_count — требует проверки кода, есть
  необъяснённые случаи в логах.
- Механизм прунинга user_portrait — не спроектирован до конца.
- Mayor как полноценный командный центр (выполнение админ-действий,
  создание структуры через диалог) — отдельная будущая стадия, не активная
  работа сейчас.
